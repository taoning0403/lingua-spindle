# LinguaSpindle v0.3.1 release notes

v0.3.1 hardens the optional HTTP/runtime service for retrying service-to-service callers. It
retains the v0.3.0 headless, library-first TXT/EPUB/manga core and adds durable idempotency, active
Job coalescing, and request correlation above that pure boundary.

Exact commands, platforms, package artifacts, checksums, and acceptance outcomes belong in
`acceptance/v0.3.1/`. Publication and deployment remain separately authorized actions.

## Highlights

- `Idempotency-Key` support for Project/Profile/Job creation, selected translation, rebuild, and
  retry, with compatibility and opt-in required modes.
- Durable hashed-key records and versioned semantic fingerprints; raw keys and Provider secrets
  are excluded from persistence and managed logs.
- Database-enforced coalescing of equivalent active Jobs across concurrent service instances.
- Atomic Project upload publication without duplicate Projects or orphan staged Artifacts.
- Conservative `indeterminate` recovery for interrupted synchronous Provider work.
- `X-Request-ID` on every success and error, plus persisted Job/Step correlation.
- Typed OpenAPI headers, 428/409 behavior, stable idempotency error codes, replay `Location`, and
  `Idempotency-Replayed`/`X-Job-Coalesced` response metadata.
- Additive migration `0004_service_idempotency.sql`; no existing Project, Job, Source, Segment, or
  Artifact is deleted or rewritten.

## Compatibility

The default `LINGUASPINDLE_REQUIRE_IDEMPOTENCY_KEY=false` preserves callers that do not yet send a
key. Set it to `true` only after every caller supplies a unique key for the covered POST
operations. Pause, resume, and cancel retain their existing natural idempotency.

The pure Python API, default dependency set, supported TXT/EPUB/image/CBZ formats, optional extras,
headless/no-GUI position, no-user-system boundary, loopback-default network posture, and external
manga Adapter separation are unchanged from v0.3.0.

## Upgrade

Stop writers and back up the complete data root or Docker Volume. First v0.3.1 runtime startup
applies migration 0004, creates the idempotency table, and adds nullable Job fingerprint/request
ID fields plus the active-Job partial unique index. Existing rows remain valid with `NULL` values.
See [the v0.3.0 to v0.3.1 migration guide](../migrations/v0.3-to-v0.3.1.md).

Rollback is restore-based. Do not run an older version against a database that has recorded
schema version 4.

## Known limits

- The optional runtime remains one host with SQLite and a local Artifact store; it is not a
  distributed queue or multi-host lock service.
- Idempotency is scoped to the six documented POST operations. It is not authentication,
  authorization, billing, or business-level deduplication.
- An interrupted synchronous Provider operation is not automatically repeated because the server
  cannot prove whether the remote side effect completed.
- Real paid Provider/model execution, real external manga model execution, external `epubcheck`,
  and unavailable platforms remain optional external tests unless explicitly recorded otherwise.
- No GUI, user system, PostgreSQL, Redis, broker, object store, or formal `novel-platform`
  dependency is introduced.

## Publication control

Local implementation and acceptance do not authorize a remote branch push, merge, `v0.3.1` tag,
GitHub Release, Wheel/image publication, or server deployment. Those actions require separate
approval after the checksummed acceptance report is reviewed.
