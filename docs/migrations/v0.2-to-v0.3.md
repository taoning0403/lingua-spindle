# Migrating v0.2.0 local data to v0.3.0

v0.3.0 does not require a new data root and does not delete old novel or manga records. The pure
library has no database; this guide applies only when the optional local runtime is installed and
an existing v0.2.0 data root is reused.

## What changes

Package migration `0003_headless_core.sql` makes one additive schema change:

- adds nullable `translation_segments.segment_key`;
- adds a partial unique index on `(job_id, segment_key)` when the key is non-null.

New runtime Segment rows can persist the stable public core Segment ID in `segment_key`. Existing
v0.2.0 rows remain valid with `NULL` in that column; the runtime supplies a deterministic legacy
public key when they are read. No Project, Source, Job, Step, translation, QA, or Artifact row is
removed or rewritten. Existing TXT, EPUB, image, and manga CBZ payloads remain under their current
private Artifact keys.

Migrations remain forward-only and are applied atomically before the runtime opens for work.

## Before upgrading

1. Stop every LinguaSpindle process using the data root. Confirm no worker is writing.
2. Record the currently installed version and explicit data-root path.
3. Copy the complete stopped data root as one consistent unit. Include the SQLite database,
   `-wal`/`-shm` files if present, Artifacts, exports, logs, and cache.
4. Keep the backup read-only until the upgrade is verified.

Do not back up only SQLite or only the Artifact directory; their identities must stay consistent.

## Install the optional runtime

The default v0.3.0 package does not install SQLAlchemy or platformdirs:

```bash
python -m pip install --upgrade 'linguaspindle[runtime,cli]'
linguaspindle doctor --data-dir /path/to/existing/data
```

`LocalRuntime(Settings(...))` is the programmatic equivalent. Construction opens the database and
applies pending migrations, but does not start a `JobRunner` thread.

If the HTTP service is used, install the server extra as well:

```bash
python -m pip install --upgrade 'linguaspindle[server,cli]'
linguaspindle serve --data-dir /path/to/existing/data
```

## Verify after migration

- `doctor` reports the expected data root and a healthy database.
- Existing novel and manga Projects, Jobs, Sources, and Artifacts are still listed.
- A representative old Artifact can be copied or downloaded and its checksum matches the stored
  metadata.
- Old completed outputs remain readable; no historical Job is automatically rerun.
- A new TXT/EPUB Job exposes stable public `segment_id` values and completes through the same
  runtime.
- A new image/CBZ Job completes with the Mock Manga Adapter before configuring a real service.

The removal of the browser GUI does not delete persistent data. Inspect it with the optional CLI,
headless HTTP API, or `LocalRuntime`.

## Rollback

There is no in-place schema downgrade. To return to v0.2.0:

1. Stop v0.3.0 completely.
2. Move the migrated data root aside for diagnosis.
3. Restore the complete pre-upgrade backup to its original location.
4. Reinstall v0.2.0 and start it against only that restored copy.

Never run v0.2.0 against a database that has recorded schema migration 0003. Do not try to remove
the column or migration marker manually.

## Application integrations

Callers that used only the v0.2.0 HTTP/CLI runtime can keep the same instance-scoped Project/Job/
Artifact data model after installing the appropriate extras. Callers embedding LinguaSpindle
should migrate new work to the public types and functions in [Python library API](../library-api.md).
The public core does not expose SQLAlchemy models or private Artifact storage paths.
