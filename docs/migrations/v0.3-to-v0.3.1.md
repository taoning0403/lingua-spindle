# Migrating v0.3.0 local data to v0.3.1

This guide applies only to installations using the optional local runtime. Core-only users have
no managed database or data root.

## What changes

Migration `0004_service_idempotency.sql` is additive:

- adds nullable `jobs.execution_fingerprint` and `jobs.request_id` columns;
- adds a partial unique index permitting one equivalent active Job in `queued`, `running`,
  `paused`, or `cancelling`; and
- creates `idempotency_records`, unique by operation scope and SHA-256 key hash, with durable
  processing/completed/failed/indeterminate state and safe resource references.

Existing Projects, Sources, Jobs, Steps, Segments, Profiles, QA rows, Artifacts, and payload bytes
are not deleted or rewritten. Existing Jobs keep `NULL` fingerprints/request IDs and are not
retroactively coalesced.

## Before upgrading

1. Stop the HTTP service, worker, CLI writers, and every process using the data root.
2. Record the installed version and exact data-root or Docker Volume identity.
3. Back up the complete stopped root as one unit, including SQLite/WAL/SHM files and all Artifact
   payloads.
4. Keep the backup read-only until v0.3.1 is verified.

Do not copy only SQLite or only Artifacts. Their identities and transaction point must match.

## Upgrade and verify

Install the v0.3.1 runtime/server package or build the exact approved image, then start it against
the existing root. Migration application is atomic and occurs before normal runtime use.

Verify that:

- `doctor` and `/health` report a healthy database;
- schema migration 0004 is recorded exactly once;
- existing Projects, Jobs, and representative Artifact checksums are unchanged;
- compatibility mode accepts a covered POST without `Idempotency-Key`;
- required mode (`LINGUASPINDLE_REQUIRE_IDEMPOTENCY_KEY=true`) returns 428 when the key is missing;
- same-key/same-request replay returns the retained resource without duplicate work; and
- every HTTP success/error includes `X-Request-ID`.

Do not perform a paid Provider call solely to validate migration. The acceptance suite uses Mock
Providers and controlled failure doubles.

## Rollback

There is no in-place downgrade. To return to v0.3.0:

1. stop v0.3.1 completely;
2. move the migrated data root aside for diagnosis;
3. restore the complete pre-upgrade backup; and
4. start v0.3.0 only against that restored copy.

Never run v0.3.0 against a database that has recorded migration 0004, and do not manually drop
the new table, columns, index, or migration marker.
